The B+ Tree

From MySQL to NTFS, this structure keeps the world's data organized.

1970: The Spinning Disk Problem

Imagine computing in 1970. We were transitioning from magnetic tapes to spinning hard disks. While disks allowed random access, they had a physical flaw: Seek Time.

Every time you requested data, a mechanical arm had to physically move across the platter. This took millisecondsâ€”eternity for a CPU.

The Challenge: How do we find one record among millions with the absolute minimum number of arm movements?

Advantages & Disadvantages

Predictable Speed

The B+ Tree is perfectly balanced. Searching for record #1 takes the exact same time as record #1,000,000. No lopsided branches.

The Range King

Thanks to the linked leaves, queries like SELECT * WHERE ID > 50 are incredibly fast. Just find 50 and scan forward.

Disk Friendly

Nodes are tuned to disk page sizes (e.g., 4KB). This minimizes physical I/O operations, which is the bottleneck of databases.

Wasted Space

Nodes are often 30-50% empty to leave room for future insertions. This is called "Internal Fragmentation".

The Write Penalty

Inserting data isn't free. If a node overflows, the split ripples up the tree. This "Write Amplification" can be heavy for write-intense apps.

The Future: LSM Trees

B+ Trees are great for reading, but what about writing massive data (like social media logs)?

Enter the LSM Tree (Log-Structured Merge). Used by Cassandra and RocksDB.

Instead of carefully placing every book on the shelf (B+ Tree), the LSM Tree just tosses new data into a pile and sorts it later in the background.
